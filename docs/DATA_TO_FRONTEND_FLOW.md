# 从数据库表到前端联调：整条链路怎么串起来

> 帮你把「建表 → 表之间的关联 → 后端查什么、接口返回什么 → 前端怎么用」串成一条线。

---

## 一、先记住一张总图

```
┌──────────────────────────────────────────────────────────────────────────┐
│  1. 建表（Flyway）                                                        │
│     SQL 脚本 → 应用启动时执行 → PostgreSQL 里真实出现 sys_user、sys_role 等  │
└──────────────────────────────────────────────────────────────────────────┘
                                        ↓
┌──────────────────────────────────────────────────────────────────────────┐
│  2. 表与表的关系（关联）                                                   │
│     用户 ←→ 角色 ←→ 菜单  靠「中间表」sys_user_role、sys_role_menu 连起来   │
└──────────────────────────────────────────────────────────────────────────┘
                                        ↓
┌──────────────────────────────────────────────────────────────────────────┐
│  3. 后端：用 Mapper 查库，用 Service 组装，用 Controller 暴露接口            │
│     接口返回 JSON：{ "user": {...}, "menus": [...] }                       │
└──────────────────────────────────────────────────────────────────────────┘
                                        ↓
┌──────────────────────────────────────────────────────────────────────────┐
│  4. 前端联调                                                               │
│     调接口拿数据 → 根据 menus 渲染侧边栏/路由 → 根据权限控制按钮/页面        │
└──────────────────────────────────────────────────────────────────────────┘
```

下面按这四步拆开讲。

---

## 二、第一步：表是怎么「创建」出来的

### 2.1 谁在建表？

- **不是**手动画个表、也不是在 IDEA 里点「建表」。
- **是**：把建表语句写在 **Flyway 的 SQL 文件**里，应用**第一次启动**时，Flyway 按顺序执行这些 SQL，表就出现在 PostgreSQL 里。

### 2.2 文件在哪、怎么执行？

- 路径：`src/main/resources/db/migration/`
- 命名：`V1__xxx.sql`、`V2__yyy.sql`（V 后面是版本号，只执行一次）
- 流程：启动 Spring Boot → 连上数据库 → Flyway 检查 `flyway_schema_history` 表里有没有 V1、V2… → **没执行过的就执行**，执行过的跳过。

所以：**表结构 = 你写的 SQL**。改表就新写一个 `V3__alter_xxx.sql`，不要直接去库里改（否则别人拉代码、重新起库时结构会乱）。

### 2.3 小结

| 你做的事           | 结果                     |
|--------------------|--------------------------|
| 写 V1__create_sys_user.sql | 库里有了 sys_user 表       |
| 写 V2__create_sys_role_menu_file.sql | 库里有了 sys_role、sys_menu、sys_user_role、sys_role_menu、sys_file |

**表创建**就这一条线：**SQL 文件 → Flyway 执行 → 数据库里真有表**。

## 三、第二步：表之间的「关联」是什么

### 3.1 为什么需要关联？

- **用户**需要**角色**（管理员、普通用户…）
- **角色**需要**菜单/权限**（能看哪些菜单、能点哪些按钮）
- 一个用户可以有多个角色，一个角色可以对应多个菜单 → 这是**多对多**关系。

多对多不能只靠「在用户表里写死角色名」，因为角色会增删改；所以用**中间表**把「用户」和「角色」连起来。

### 3.2 我们项目里的关系（画成表）

```
sys_user (用户)           sys_user_role (用户-角色)      sys_role (角色)
─────────────────        ─────────────────────────      ─────────────────
id                        user_id  ──────────────────→  id
username                  role_id  ──────────────────→  id
...                       (一个 user_id + role_id 就是「某用户拥有某角色」)

sys_role (角色)           sys_role_menu (角色-菜单)      sys_menu (菜单)
─────────────────        ─────────────────────────      ─────────────────
id                        role_id  ──────────────────→  id
code                      menu_id  ──────────────────→  id
                          (一个 role_id + menu_id 就是「某角色能看到某菜单」)
```

- **sys_user_role**：存的是「哪个用户(id) 拥有 哪个角色(id)」。
- **sys_role_menu**：存的是「哪个角色(id) 拥有 哪个菜单(id)」。

在数据库里，这种关联用**外键**（如 `user_id REFERENCES sys_user(id)`）保证：不会出现「不存在的用户 id」或「不存在的角色 id」。

### 3.3 在代码里「关联」怎么用？

关联**不是**在表里自动「带出」另一张表的数据，而是**你写查询**把多张表连起来用：

- 例：**查「当前用户能看到哪些菜单」**：
  1. 先查 **sys_user_role**：当前用户的 `user_id` → 得到一堆 `role_id`。
  2. 再查 **sys_role_menu**：这些 `role_id` → 得到一堆 `menu_id`。
  3. 再查 **sys_menu**：这些 `menu_id` → 得到菜单的 name、path、permission 等。

所以：**表之间的关联 = 通过中间表的 id 把多张表的数据拼起来**。拼的逻辑写在 **Service** 里（调多个 Mapper），或写在一个 Mapper 的 SQL 里（JOIN 多张表）。

### 3.4 小结

| 概念           | 在数据库里                     | 在代码里                           |
|----------------|--------------------------------|------------------------------------|
| 用户-角色      | sys_user_role 存 user_id、role_id | Service 先查 UserRole，再查 Role   |
| 角色-菜单      | sys_role_menu 存 role_id、menu_id | Service 先查 RoleMenu，再按 menu_id 查 Menu |
| 「当前用户的菜单」 | 无单独表，靠上面三张表算出来   | 一次查 user_role → role_menu → menu，组装成列表/树 |

## 四、第三步：后端怎么「查关联」并变成接口

### 4.1 谁在查库？谁在组装？

- **Mapper**：只负责「单表或固定 JOIN」的 SQL，返回实体（User、Role、Menu、UserRole、RoleMenu）。
- **Service**：负责**业务逻辑**，比如「根据 userId 查出这个人能看到的菜单」：
  - 调 `UserRoleMapper.findByUserId(userId)` → 得到 roleId 列表
  - 再对每个 roleId 调 `RoleMenuMapper.findByRoleId(roleId)` → 得到 menuId 列表（去重）
  - 再根据 menuId 调 `MenuMapper.findById` 或一次查 `MenuMapper.findByIds` → 得到 Menu 列表
  - 最后可以再组装成**树形**（parent_id 建父子关系）给前端
- **Controller**：只调 Service，把 Service 返回的 Java 对象**转成 JSON** 通过 HTTP 返回。

所以：**关联的体现 = 在 Service 里按「用户 → 角色 → 菜单」的顺序查多张表，组装成前端要的结构**。

### 4.2 接口返回什么、前端会拿到什么？

典型两个接口：

1. **登录接口**  
   - 入参：`username`、`password`  
   - 校验通过后：查用户、查该用户的角色（可能再查权限），生成 **JWT**。  
   - 返回给前端：`{ "token": "xxx", "user": { "id", "username", "roles": ["ADMIN"] } }`  
   - 前端：把 token 存起来（如 localStorage），后面请求都在 Header 里带这个 token。

2. **获取当前用户菜单/权限**  
   - 入参：无（或只靠 Header 里的 token 识别是谁）  
   - 后端：从 token 里取出 userId → 按上面说的「用户 → 角色 → 菜单」查一遍 → 组装成树形。  
   - 返回：`{ "menus": [ { "id", "name", "path", "children": [...] } ] }`  
   - 前端：用这份 menus 动态生成**路由**和**侧边栏**；用里面的 `permission` 控制**按钮**显隐（如没有 `user:delete` 就不显示删除按钮）。

所以：**前后端联调 = 前端调这两个（以及后续的 CRUD）接口，拿到的就是「用户信息 + 菜单树 + 权限」**，前端只消费 JSON，不关心你库里几张表、怎么 JOIN。

## 五、第四步：前端怎么「联调」

### 5.1 联调在调什么？

- **调**：前端的请求（如 axios）发到后端的 **URL**（如 `POST /api/auth/login`、`GET /api/menus`）。
- **调通**：后端返回 200 和约定好的 JSON 结构，前端能解析并用到界面（登录后跳转、侧边栏出菜单、按钮按权限显示）。

### 5.2 前端会用到哪些数据？

| 后端提供（接口）     | 前端用法 |
|----------------------|----------|
| 登录接口返回 token、user | 存 token；存 user 用于显示昵称、头像等 |
| 菜单树接口返回 menus   | 根据 path 注册路由；根据 name/path 渲染侧边栏；根据 permission 做按钮权限 |
| 用户列表、角色列表等   | 表格展示、表单下拉等 |

### 5.3 和「表、关联」的关系

- **表、关联**都在**后端**：数据库里是 sys_user、sys_role、sys_menu、sys_user_role、sys_role_menu；后端用 Mapper 查、Service 组装。
- **前端**只认**接口和 JSON**：不关心你有几张表、有没有中间表。只要接口稳定（URL、入参、返回结构约定好），联调就是「前端发请求 → 后端按表与关联查数据 → 返回 JSON → 前端渲染」。

所以：**表与关联是为「接口能返回正确数据」服务的；前端只和接口契约打交道。**

## 六、用一条「查当前用户菜单」的线串起来

1. **数据库**：已有表 sys_user、sys_role、sys_menu、sys_user_role、sys_role_menu（Flyway 建好）。
2. **关联**：用户 ↔ 角色 靠 sys_user_role；角色 ↔ 菜单 靠 sys_role_menu。
3. **后端**：  
   - Controller 提供 `GET /api/menus`（或 /api/user/menus），从 token 里取 userId。  
   - Service：userId → UserRoleMapper.findByUserId → roleIds → RoleMenuMapper.findByRoleId → menuIds → MenuMapper 查菜单 → 组装树。  
   - 返回 `{ "menus": [ ... ] }`。
4. **前端**：请求 `GET /api/menus`，拿到的 menus 用来生成路由和侧边栏；联调时确认 URL、Header（带 token）、返回格式即可。

整条链就是：**建表（Flyway）→ 表之间用中间表关联 → 后端按关联查库、组装 JSON → 前端调接口、用 JSON 渲染**。你只要记住：**表/关联在后端，接口是前后端的唯一交界，前端不直接碰数据库。**
