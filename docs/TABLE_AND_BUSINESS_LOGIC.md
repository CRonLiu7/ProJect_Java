# 当前项目的建表逻辑与业务逻辑说明

> 结合本项目已有的表（sys_*）和规划中的业务（审批、订单商品），说明：**表为什么这样建**、**业务是怎么跑的**、**代码和表/业务怎么对应**。看代码时可以把本文当「说明书」对照。

---

## 一、本项目在做什么「业务」？（先建立整体印象）

本项目是一个 **后台管理系统**，主要包含三块业务：

| 业务         | 一句话说明 |
|--------------|------------|
| **权限管理** | 谁可以登录、登录后能看到哪些菜单、能点哪些按钮（如「删除用户」） |
| **审批流**   | 某类业务（如订单）需要「提交 → 某人审批 → 通过/驳回」，流程可配置 |
| **订单与商品** | 维护商品、用户下单、订单里有多个商品；订单可以走审批 |

这三块会用到不同的表，但**权限是基础**：所有操作都先要「能登录、有权限」。下面先讲**建表逻辑**，再讲**每块业务怎么跑、对应哪些表**。

---

## 二、建表逻辑：为什么有这些表、字段为什么这样设计

### 2.1 表的分组与建表顺序

- **sys_** 开头：系统级、和「谁能用系统、能看什么」相关。
- **biz_** 开头：业务级，如订单、商品、审批（后续阶段会建）。

**建表顺序**（由 Flyway 的 V1、V2 决定）：

1. 先有 **sys_user**（用户）：没有用户就没有「谁在操作系统」。
2. 再有 **sys_role**（角色）、**sys_menu**（菜单）：角色和菜单是「权限」的两大要素。
3. 然后 **sys_user_role**、**sys_role_menu**：把「用户」和「角色」、「角色」和「菜单」连起来（多对多必须用中间表）。
4. 最后 **sys_file**：和用户有关（谁上传的），所以放在 sys_user 之后。

外键依赖决定了顺序：sys_user_role 依赖 sys_user 和 sys_role，所以必须先有 user 和 role 表。

---

### 2.2 每张表在「业务里」是干什么的

#### sys_user（用户表）

| 字段           | 含义 | 设计理由 |
|----------------|------|----------|
| id             | 主键 | 唯一标识一个用户，别的表用 user_id 引用 |
| username       | 登录名 | 登录时输入，唯一（uk_sys_user_username） |
| password_hash  | 密码哈希 | 不能存明文，用 BCrypt 等加密后存这里 |
| created_at / updated_at | 创建、更新时间 | 审计、排查问题用 |

**业务含义**：这张表回答「系统里有哪些人」。登录时用 username + 密码（加密后和 password_hash 比对）判断是不是本人。

---

#### sys_role（角色表）

| 字段   | 含义 | 设计理由 |
|--------|------|----------|
| id     | 主键 | 别的表用 role_id 引用 |
| name   | 角色名称 | 给人看的，如「管理员」「普通用户」 |
| code   | 角色编码 | 给程序用的，如 ADMIN、USER；判断权限时用 code，不用 name |

**业务含义**：角色是「权限的集合」。一个人通过「拥有某个角色」间接拥有这个角色下的所有菜单和按钮权限。code 唯一（uk_sys_role_code），程序里写死判断 `if (user.hasRole("ADMIN"))` 这类逻辑。

---

#### sys_menu（菜单/权限表）

| 字段      | 含义 | 设计理由 |
|-----------|------|----------|
| id        | 主键 | 被 sys_role_menu 引用 |
| parent_id | 父菜单 id | 0 表示顶级，否则是某个菜单的子项 → 形成树形菜单 |
| name      | 显示名称 | 侧边栏上显示「用户管理」「角色管理」 |
| path      | 前端路由 | 前端用 path 做路由，如 /system/user |
| permission | 权限标识 | 如 user:list、user:delete，前端据此控制按钮显隐、后端也可做接口权限 |
| type      | 类型 | 1=菜单（占一个侧边栏项）、2=按钮（只对应一个操作权限） |
| sort_order | 排序 | 菜单展示顺序，数字越小越靠前 |

**业务含义**：既描述「菜单长什么样」（name、path、树形），又描述「要有什么权限才能看/点」（permission、type）。前端根据菜单树渲染侧边栏和路由，根据 permission 控制按钮。

---

#### sys_user_role（用户-角色，中间表）

| 字段     | 含义 |
|----------|------|
| user_id  | 用户 id |
| role_id  | 角色 id |
| 唯一约束 | (user_id, role_id) 同一人同一角色只一条 |

**业务含义**：记录「谁拥有哪个角色」。一个用户可以有多个角色（如既是管理员又挂一个「审批人」），一个角色也可以给多个人。所以用户和角色是**多对多**，必须用中间表存 (user_id, role_id)。

**建表逻辑**：外键指向 sys_user 和 sys_role，ON DELETE CASCADE 表示删用户时顺带删掉他的所有角色关联。

---

#### sys_role_menu（角色-菜单，中间表）

| 字段    | 含义 |
|---------|------|
| role_id | 角色 id |
| menu_id | 菜单 id |
| 唯一约束 | (role_id, menu_id) |

**业务含义**：记录「哪个角色能看/用哪个菜单（或按钮）」。查「当前用户能看到哪些菜单」时：先查 sys_user_role 得到该用户的 role_id 列表，再查 sys_role_menu 得到这些角色下的 menu_id，再去 sys_menu 取菜单详情并拼成树。

---

#### sys_file（文件记录表）

| 字段          | 含义 |
|---------------|------|
| original_name | 用户上传时的文件名 |
| storage_path  | 文件在服务器或 MinIO 上的存储路径，下载时用 |
| file_size     | 文件大小 |
| uploader_id   | 上传人（关联 sys_user.id） |

**业务含义**：上传文件时，文件实体存磁盘或 MinIO，只在数据库里留一条「记录」，方便按 id 下载、按上传人查列表。uploader_id 表示「谁传的」，便于权限和审计。

---

### 2.3 建表逻辑小结（对应到代码）

- **Flyway 脚本**（V1、V2）＝ 建表 + 初始数据的「唯一真相」；改表就加新版本脚本，不要直接改库。
- **实体类**（User、Role、Menu…）＝ 和表一一对应，字段对应列，供 MyBatis 映射。
- **Mapper**＝ 按表查/改/删，返回实体或列表。**关联不写在表里，写在查法里**：例如「查用户的菜单」要查 user_role → role_menu → menu，在 Service 里调多个 Mapper 或一个 Mapper 里写 JOIN。

## 三、业务逻辑一：权限管理（已实现的表都围绕这个）

### 3.1 业务要解决的问题

- 谁可以登录？→ 只有 sys_user 里存在、且密码对的人。
- 登录后能看到哪些菜单？→ 由该用户拥有的**角色**，以及这些角色关联的**菜单**决定。
- 某个按钮（如「删除用户」）能不能点？→ 看该用户是否拥有带 `user:delete` 权限的菜单/按钮（type=2 的 menu）。

### 3.2 数据流（表怎么参与）

1. **登录**  
   - 输入：username、password。  
   - 后端：用 username 查 sys_user，用 BCrypt 校验 password 和 password_hash。  
   - 通过后：查 sys_user_role 得到该用户的 role_id 列表，可再查 sys_role 得到角色 code/name；生成 JWT（里可带 userId、roles），返回给前端。

2. **获取「当前用户可看的菜单」**  
   - 输入：当前用户 id（从 JWT 里取）。  
   - 步骤：  
     - sys_user_role  where user_id = ? → 得到 role_id 列表；  
     - sys_role_menu  where role_id in (…) → 得到 menu_id 列表（去重）；  
     - sys_menu  where id in (…) → 得到菜单完整信息；  
     - 在内存里按 parent_id 拼成树（parent_id=0 为根，再递归 children）。  
   - 返回：树形菜单给前端，前端用 path 做路由、用 permission 做按钮权限。

3. **接口权限（后端）**  
   - 例如：删除用户接口要求有 `user:delete`。  
   - 后端从 JWT 取出 userId → 按上面步骤查出该用户所有 menu 的 permission（或只查 type=2 的），看是否包含 `user:delete`；没有则 403。

### 3.3 和代码的对应关系

- **Controller**：只做「接收登录请求、调登录 Service」「接收获取菜单请求、调菜单 Service」。
- **Service**：  
  - 登录：UserMapper 查用户 → 校验密码 → UserRoleMapper 查角色 → 发 JWT。  
  - 菜单：UserRoleMapper 按 userId 查 → RoleMenuMapper 按 roleId 查 → MenuMapper 查菜单 → 拼树。  
- **Mapper/XML**：只做单表或简单 JOIN，不包含「拼树」等业务，业务在 Service。

所以：**权限业务逻辑 = 在 Service 里按「用户 → 角色 → 菜单」的顺序查表、拼树、判断 permission」**；表结构（user、role、menu、user_role、role_menu）就是为这套逻辑服务的。

## 四、业务逻辑二：审批流（表后续会建，逻辑先讲清）

### 4.1 业务要解决的问题

- 某类业务（如「订单超过 1 万要审批」）需要走流程：提交 → 指定的人审批 → 通过或驳回。
- 流程可能有多级、多人，所以要「流程定义」「一次审批实例」「每个节点上的待办任务」。

### 4.2 表与业务含义（规划）

| 表名                       | 业务含义 |
|----------------------------|----------|
| biz_approval_definition    | 流程定义：叫什么名字、关联哪种业务（如订单）、有哪些节点（谁批、顺序）。 |
| biz_approval_instance      | 一次审批实例：对应「某一条业务数据」（如某笔订单），当前到哪个节点、状态（进行中/通过/驳回）。 |
| biz_approval_task          | 具体任务：谁在哪个节点、待办还是已办、批注。 |

**业务流**：  
1. 用户提交订单（或其它业务）→ 若需审批，则创建一条 approval_instance，并生成第一个节点的 approval_task（指定审批人）。  
2. 审批人登录后看到「待办」→ 点通过/驳回 → 更新 instance 状态，可能生成下一节点 task 或结束。  
3. 查询「我的待办」＝ 查 approval_task 里 assignee=当前用户且未完成的。

建表时会把 instance 和业务表（如 biz_order）通过 business_type + business_id 或 order_id 关联起来，这样「订单」和「审批」就串起来了。

## 五、业务逻辑三：订单与商品（表后续会建）

### 5.1 业务要解决的问题

- 维护商品（名称、编码、价格、库存等）。
- 用户下单：生成订单头（总金额、状态、下单人）+ 订单明细（商品、数量、单价、小计）。
- 订单可能要走审批：订单表里会有一个字段关联 biz_approval_instance（如 approval_instance_id），表示「这笔单子对应的审批」。

### 5.2 表与业务含义（规划）

| 表名                | 业务含义 |
|---------------------|----------|
| biz_product         | 商品主数据。 |
| biz_order           | 订单头：单号、用户、总金额、状态（待审批/已通过/已发货等）、若走审批则存 approval_instance_id。 |
| biz_order_item      | 订单明细：order_id、product_id、数量、单价、小计。 |

**业务流**：  
- 下单：插一条 order，再插多条 order_item；若需审批则再建 approval_instance 和 task。  
- 查订单：order 为主，order_item 按 order_id 查出来一起返回。  
- 审批通过后：把 order 状态改为「已通过」，后续可再改「已发货」等。

建表逻辑：order 和 product 通过 order_item 关联（订单里包含哪些商品）；order 和审批通过 approval_instance_id 关联。

## 六、建表逻辑 + 业务逻辑 如何对应到「看代码」

- **建表逻辑**：看 Flyway 的 SQL（V1、V2）和本表说明 → 知道「每张表、每个字段是干什么的、为什么有中间表、外键是谁」。
- **业务逻辑**：看上面三块业务（权限、审批、订单）的数据流 → 知道「先查哪张表、再查哪张表、在哪儿拼树、在哪儿改状态」。
- **写代码时**：  
  - **Mapper 层**：只关心「按表查/插/改」，对应「建表」出来的那几张表。  
  - **Service 层**：按「业务逻辑」的顺序调 Mapper（例如先 UserRole 再 RoleMenu 再 Menu），并做拼树、判权限、改状态。  
  - **Controller 层**：只接收请求、调 Service、返回 JSON。

这样，**建表 = 为业务提供数据结构；业务逻辑 = 在 Service 里按业务顺序使用这些表**。看代码时只要分清「这段是在查哪张表、在为哪块业务服务」，就容易串起来了。
