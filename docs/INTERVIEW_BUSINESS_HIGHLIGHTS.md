# 面试可讲的业务流与加分项

> 当前项目已有：权限（RBAC）、登录 JWT、菜单树。下面补充一批**有辨识度、面试好讲**的核心业务流和技术点，并给出「怎么设计、怎么讲」。  
> **这些业务已纳入整体阶段计划**，详见 → **[ARCHITECTURE_PLAN.md](./ARCHITECTURE_PLAN.md)** 第七节「分阶段实施计划」；在对应阶段实现即可。

## 一、目前已有的基础（可以这样概括）

- **技术栈**：Spring Boot 3、MyBatis、PostgreSQL、Flyway、JWT、后续会加 Redis。
- **已实现**：RBAC 权限、登录/登出（JWT）、当前用户菜单树、表与关联设计（用户-角色-菜单）。
- **面试一句话**：「后台管理系统，权限用 RBAC + JWT，菜单按角色动态返回树形结构，表结构用 Flyway 管理。」

要加分，需要在「业务复杂度」和「技术痛点」上再补几块，下面按**优先级**列出来。

## 二、建议补充的核心业务流（按面试价值排）

### 1. 审批流（流程引擎 / 状态机）—— 高价值

**业务场景**  
订单/报销/请假等需要「提交 → 审批人通过/驳回 → 可能多级审批」的流程。

**技术点**

- **流程定义**：审批流程可配置（哪些节点、谁审批、顺序），存 `biz_approval_definition`。
- **流程实例**：每笔业务单对应一条实例，存当前节点、状态（进行中/通过/驳回），即 `biz_approval_instance`。
- **审批任务**：每个节点产生「待办任务」给具体人，审批后变已办，支持批注、转交，即 `biz_approval_task`。
- **状态扭转**：只有合法状态才能流转（如只有「待审批」才能点通过/驳回），用**状态机**思维，避免乱改状态。

**面试怎么讲**

- 「我们做了可配置的审批流：流程定义、实例、任务三张表，实例和业务单（如订单）关联，任务表记录待办/已办和批注。」
- 「状态扭转做了校验，避免从未审批直接改成已通过；多节点时按定义顺序推进，支持驳回上一节点或驳回到发起人。」

**实现建议**  
先做「固定两节点：提交 → 审批人」的简化版，再扩展为可配置多节点；表结构在架构计划里已有，可直接落 Flyway + Service 状态机。

---

### 2. 订单 + 状态机 + 库存（并发与一致性）—— 高价值

**业务场景**  
下单减库存、取消/超时还库存；订单有明确状态：待支付→已支付→待发货→已发货→已完成，以及取消、退款等。

**技术点**

- **订单状态机**：状态只能按既定规则流转（如待支付→已支付→待发货），在 Service 里集中判断「当前状态能否转到目标状态」，不开放随意改状态。
- **库存扣减与回滚**：下单时减库存，用**乐观锁**（version 或 stock 条件更新）防止超卖；取消/超时关单时把库存加回去，同一事务里改订单状态 + 还原库存。
- **超时关单**：下单未支付超过 N 分钟自动关单并回滚库存，可用定时任务扫「待支付且创建超过 N 分钟」的订单批量处理。

**面试怎么讲**

- 「订单做了状态机，所有状态变更在一个 Service 里统一校验，避免非法流转。」
- 「库存用乐观锁更新，下单和取消都在事务里同时改订单和库存，保证一致性；超时未支付有定时任务关单并回滚库存。」

**实现建议**  
商品表加 `stock`、订单明细表已有；订单表加 `status`、`version`（可选）；库存更新用 `UPDATE ... SET stock = stock - ? WHERE id = ? AND stock >= ?`。

### 3. 幂等（防重复提交 / 重复回调）—— 中高价值

**业务场景**  
用户连续点两次「提交订单」、支付网关重复回调，不能产生两笔订单或重复加款。

**技术点**

- **下单幂等**：前端传唯一 `clientToken`（如 UUID），后端以 `clientToken` 为唯一键，第一次创建订单并落库，第二次同一 token 直接返回已有订单，不重复扣库存。
- **支付/回调幂等**：第三方支付回调带唯一 `outTradeNo` 或 `requestId`，先查是否已处理过，已处理直接返回成功；未处理再执行业务并记录「已处理」。

**面试怎么讲**

- 「下单接口做了幂等：前端带 clientToken，后端用唯一约束或先查再插，同一 token 只落一单并只扣一次库存。」
- 「支付回调用 outTradeNo/requestId 做幂等，先查再更新，避免重复入账。」

**实现建议**  
订单表加 `client_token` 唯一索引；支付回调表或订单扩展字段记录「已回调/已入账」状态。

---

### 4. 操作日志 / 审计（谁在何时做了什么）—— 中价值

**业务场景**  
敏感操作（登录、改权限、删用户、审批通过/驳回、订单状态变更）需要留痕，便于排查与合规。

**技术点**

- **AOP 记录**：用切面拦截标注了 `@OperationLog` 的接口，从 SecurityContext 取当前用户、从参数/返回值取业务 id，写入 `sys_operation_log`（user_id、operation、target_type、target_id、request_id、created_at）。
- **不记敏感内容**：请求体里密码、token 不落库，只记操作类型与对象 id。

**面试怎么讲**

- 「关键操作用 AOP 统一记操作日志，谁、什么时间、对哪个业务对象做了啥，方便审计和排查；敏感字段不记录。」

**实现建议**  
`sys_operation_log` 表 + 一个注解 + 一个切面，先覆盖登录、审批、订单状态变更。

---

### 5. Redis：限流 + 缓存 + 登出黑名单（已规划，落地即加分）—— 中高价值

**业务场景**  
登录防暴力破解、接口限流、登出后 token 立即失效、菜单/权限缓存减轻 DB 压力。

**技术点**

- **登录限流**：按 IP 或 username 计数，失败 N 次后锁定一段时间，用 Redis key 过期。
- **接口限流**：按用户或 IP 滑动窗口/固定窗口，超限返回 429。
- **JWT 黑名单**：登出时把 token 或 jti 写入 Redis，过期时间等于 token 剩余有效期；校验时先查黑名单。
- **菜单/权限缓存**：登录后或首次查菜单时把结果缓存到 Redis，TTL 10～30 分钟，减少查库。

**面试怎么讲**

- 「登录用 Redis 做失败次数限流，登出把 token 放进黑名单实现即时失效，菜单和权限做了短 TTL 缓存。」

**实现建议**  
按架构计划阶段 4 引入 Redis，先做登录限流 + 黑名单 + 菜单缓存，再做接口限流。

---

### 6. 数据权限（本部门 / 本人）—— 可选，中价值

**业务场景**  
销售只能看自己的客户，部门经理看本部门，管理员看全部。

**技术点**

- **数据范围**：在角色或用户上挂「数据范围」：本人、本部门、本部门及子部门、全部。
- **查询拼接**：在 Mapper 层或 Service 层根据当前用户的数据范围，动态拼 `WHERE` 条件（如 `creator_id = ?` 或 `dept_id IN (?)`）。

**面试怎么讲**

- 「除了菜单权限，还做了数据权限：按角色配置数据范围，查询时根据当前用户动态加条件，实现不同人看不同数据。」

**实现建议**  
用户表或角色表加 `data_scope` 字段；Service 查列表时先算「可见范围」再传给 Mapper（或 Mapper 接收 scope 参数）。

## 四、面试时怎么串起来讲

- **项目定位**：「一个带权限、审批、订单的后台管理系统，技术栈是 Spring Boot、MyBatis、PostgreSQL、Redis。」
- **权限**：「RBAC，JWT 登录，菜单按角色动态返回，接口用 @PreAuthorize 控制。」
- **审批**：「做了审批流引擎：流程定义、实例、任务，和订单关联，状态机控制流转。」
- **订单**：「订单有状态机，库存用乐观锁，下单幂等，超时关单并回滚库存。」
- **稳定性与安全**：「登录限流、JWT 黑名单、操作日志审计；支付/下单幂等防重复。」
- **扩展**：「菜单和权限用 Redis 缓存，后续可加数据权限、更多审批节点。」

这样既有「业务复杂度」（审批、订单状态机、库存），又有「技术痛点」（幂等、并发、限流、审计），面试官容易追问，你也都有点可讲。如果你愿意，我们可以从「审批流」或「订单+库存+状态机」里选一块先细化表设计和接口列表，再落到代码。
